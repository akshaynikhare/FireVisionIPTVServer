# ADR-003: Pairing PIN Rate Limiting Strategy

## Status

Accepted

## Context

The FireVision IPTV system uses 6-digit PINs for TV device pairing. There are two distinct pairing flows:

1. **Default/shared channel list code** — A 6-character hex code (`channelListCode`) assigned to each user. This code is semi-permanent and used by the TV app to fetch playlists. The default code (set via `DEFAULT_CHANNEL_LIST_CODE` env var) is shared publicly and intended for open access.

2. **PIN-based device pairing** — A temporary 6-digit PIN generated by the TV, entered on the web dashboard to link a user account to a device. PINs expire after 10 minutes.

The 6-hex-char code space (~16M combinations) is brute-forceable without rate limiting. We need to protect user-specific codes while keeping the default code accessible.

## Decision

### Default channel list code: No rate limit

- The default code is public/shared — it's meant to be easily accessible for new users and demo purposes.
- Rate limiting the default code would degrade the onboarding experience.
- The default code only grants access to the default channel list, not user-specific data.

### User-specific channel list codes: Rate limit on lookup failures

- Apply rate limiting to `/tv/playlist/:code`, `/tv/verify/:code`, and `/tv/pair` endpoints.
- Rate limit: **10 failed code lookups per IP per 15 minutes**.
- Failed lookups (404 responses) count toward the limit; successful lookups do not.
- The default code is exempt from rate limit counting.

### PIN-based pairing endpoints: Strict rate limiting

- `POST /tv/pairing/request`: **5 requests per IP per 15 minutes** (prevents PIN flooding).
- `GET /tv/pairing/status/:pin`: **30 requests per IP per minute** (polling, needs to be lenient).
- `POST /tv/pairing/confirm`: **10 attempts per session per 15 minutes** (prevents PIN brute-force from web side).

## Alternatives Considered

1. **Increase code entropy (e.g., 8+ chars)** — Better security but worse UX for TV remote input. Rejected for now; the rate limit is sufficient protection.
2. **Rate limit everything uniformly** — Would block legitimate default-code users during high traffic. Rejected.
3. **CAPTCHA on pairing** — Not feasible on TV devices. Rejected.
4. **IP allowlisting** — Too restrictive for a consumer product. Rejected.

## Consequences

### Positive

- Default code remains frictionless for onboarding and demos.
- User-specific codes protected against brute-force enumeration.
- PIN-based pairing protected against flooding and guessing.
- Audit logging on all pairing events provides visibility.

### Negative

- Sophisticated attackers using distributed IPs can still attempt slow brute-force. Acceptable risk given the 6-char code space and the fact that codes can be revoked.
- Rate limit state requires Redis (already in use for sessions/cache).

### Implementation Notes

- Use `express-rate-limit` with Redis store for distributed rate limiting.
- Exempt the default code by checking against `process.env.DEFAULT_CHANNEL_LIST_CODE` in the rate limit skip function.
- Log rate limit hits to audit log for monitoring.
